## 1. Для кого этот roadmap

Kali Linux — **не первый** дистрибутив для знакомства с Linux и **не** система для повседневной работы. Браться за него стоит, когда вы уже:

- уверенно чувствуете себя в терминале (навигация, права, пакеты, процессы);
- понимаете основы сетей (IP, порты, TCP/UDP, DNS, HTTP);
- готовы много читать документацию и не бояться ломать систему в виртуалке.

Если этой базы нет — сначала пройдите общий курс по Linux и сетям, иначе инструменты Kali останутся «чёрным ящиком».

**Как пользоваться:** идите сверху вниз, не перепрыгивайте. Заводите конспект, записывайте команды и выводы своими словами. Делайте снапшоты виртуалки перед экспериментами.

## 2. Фундамент

Чек-лист знаний, без которых дальше будет тяжело:

- **Linux:** `ls`, `cd`, `cp`, `mv`, права (`chmod`/`chown`), `sudo`, пакеты (`apt`), процессы (`ps`, `kill`), `systemctl`, логи (`journalctl`).
- **Сети:** что такое IP-адрес и подсеть, порт, разница TCP/UDP, как работает DNS, HTTP-запрос и ответ, что такое NAT.
- **Скриптинг:** базовый **Bash** (переменные, циклы, пайпы) и желательно **Python** — большинство инструментов и эксплойтов написаны на нём.

**Задание:** в обычной виртуалке (Ubuntu/Debian) напишите bash-скрипт, который пингует список IP из файла и выводит, какие хосты живые. Это разминка перед сетевой разведкой.

## 3. Установка и первая настройка

### Откуда брать

Скачивайте образы **только** с официального сайта `kali.org/get-kali` и обязательно сверяйте контрольную сумму **SHA256** — это защита от подменённого образа.

### Способы установки

- **Виртуалка (рекомендуется для учёбы)** — VirtualBox или VMware. Kali публикует готовые образы для обеих. Полная изоляция, легко откатиться через снапшот.
- **Live-USB** — загрузка с флешки без установки; режим *persistence* сохраняет файлы между запусками.
- **WSL2 (Windows)** — пакет `kali-linux` из Microsoft Store; удобно для консольных инструментов.
- **На «голое железо»** — для повседневки не нужно; имеет смысл только для специфических задач (например, работа с Wi-Fi-адаптером напрямую).

### Первые шаги после установки

```bash
# Полное обновление (Kali — rolling release, это важно делать регулярно)
sudo apt update && sudo apt full-upgrade -y

# Гостевые дополнения VirtualBox (общий буфер, нормальное разрешение экрана)
sudo apt install -y virtualbox-guest-x11 && sudo reboot

# Сменить дефолтный пароль обычного пользователя
passwd

# Утилита настройки самой Kali (метапакеты, оболочка, undercover-режим)
sudo apt install -y kali-tweaks && kali-tweaks
```

### Метапакеты — ставьте только нужное

Полный Kali тяжёлый. Разумнее ставить группы инструментов под задачу:

- `kali-linux-headless` — без графики, только консоль (для VPS/серверов).
- `kali-linux-default` — стандартный набор из установщика.
- `kali-linux-large` — расширенный набор.
- `kali-tools-web`, `kali-tools-wireless`, `kali-tools-forensics` и др. — точечные группы.

```bash
apt search kali-tools          # посмотреть все группы
sudo apt install -y kali-tools-web
```

### Гигиена

- **Не работайте под root постоянно** — современная Kali создаёт обычного пользователя, так и оставьте, права поднимайте через `sudo`.
- **Снапшот** делайте сразу после чистой настройки.
- **Обновляйтесь регулярно** — rolling release быстро устаревает.

**Задание:** установите Kali в VirtualBox, сделайте полное обновление, поставьте гостевые дополнения, смените пароль и сделайте снапшот «clean install». Запишите в конспект шаги, чтобы повторить за 10 минут.

## 4. Сборка домашней лаборатории

Без своей лаборатории легально практиковаться негде. Минимальная схема — две и более виртуалки в **изолированной** сети (Host-Only / Internal Network в VirtualBox), чтобы трафик не выходил в реальную сеть.

**Мишени для атак (легально, ставятся в свою же сеть):**

- **Metasploitable 2/3** — намеренно уязвимая Linux-машина.
- **DVWA**, **OWASP Juice Shop**, **bWAPP** — уязвимые веб-приложения.
- **VulnHub** — образы уязвимых ВМ для скачивания.

**Схема сети:**

```
[ Kali (атакующий) ] --- внутренняя сеть 192.168.56.0/24 --- [ Metasploitable / DVWA (мишени) ]
                          (нет выхода в интернет и в домашнюю сеть)
```

**Задание:** разверните Kali и Metasploitable 2 в одной внутренней сети VirtualBox. Убедитесь, что они пингуют друг друга, но мишень **не** доступна из вашей домашней сети. Сделайте снапшоты обеих машин.

## 5. Инструментарий по этапам

Карта основных инструментов Kali, сгруппированных по этапам работы. Изучайте по одному, читая `man` и официальную документацию.

### Разведка (Reconnaissance)

- **nmap** — сканер портов и сервисов, основа основ.
- **theHarvester**, **recon-ng** — сбор открытой информации (OSINT).
- **dnsenum**, **dnsrecon** — разведка DNS.

### Сканирование уязвимостей

- **nmap NSE-скрипты** (`--script vuln`), **nikto** (веб-сервер), **wpscan** (WordPress).

### Эксплуатация

- **Metasploit Framework** — фреймворк для проверки уязвимостей.
- **searchsploit** (Exploit-DB) — поиск готовых эксплойтов.

### Веб

- **Burp Suite** (Community), **OWASP ZAP** — перехват и анализ HTTP.
- **gobuster**, **ffuf** — перебор директорий и файлов.

### Беспроводные сети

- **aircrack-ng**, **kismet** — анализ Wi-Fi (только своей сети!).

### Пароли

- **john** (John the Ripper), **hashcat** — восстановление паролей по хешам.
- **hydra** — проверка стойкости аутентификации сервисов.

**Пример разведки (только в своей лаборатории):**

```bash
# Найти живые хосты в своей подсети
nmap -sn 192.168.56.0/24

# Порты, версии сервисов и базовые скрипты на своей мишени
nmap -sV -sC -p- 192.168.56.101

# Проверка веб-сервера мишени
nikto -h http://192.168.56.101
```

**Задание:** просканируйте свою мишень через `nmap -sV`, выпишите все открытые порты и версии сервисов. Для каждого устаревшего сервиса найдите в `searchsploit`, есть ли известные проблемы, и опишите, **как администратор это закрыл бы**.

## 6. Методология

Хаотичный «потыкать инструменты» — это не пентест. Работа идёт по этапам:

1. **Разведка (Recon)** — сбор информации о цели. Пассивная (без касания цели) и активная.
2. **Сканирование (Scanning)** — живые хосты, открытые порты, версии, уязвимости.
3. **Получение доступа (Exploitation)** — проверка найденных уязвимостей на учебной мишени.
4. **Закрепление и пост-эксплуатация** — оценка, что злоумышленник смог бы сделать дальше.
5. **Заметание следов** — для оценки, насколько атака заметна (в учебных целях).
6. **Отчёт (Reporting)** — **самый важный этап**: что нашли, как воспроизвести, как починить.

Без отчёта пентест бесполезен. Учитесь думать как защитник: каждая находка → риск → конкретная рекомендация по устранению.

**Задание:** пройдите свою мишень по всем этапам и оформите мини-отчёт по шаблону «находка → доказательство → риск → рекомендация».

## 7. Веб-безопасность

Большая часть реальных задач — это веб. Базовый ориентир — **OWASP Top 10** (самые частые классы уязвимостей: инъекции, сломанная аутентификация, XSS, небезопасная конфигурация и др.).

**Связка инструментов:** браузер + **Burp Suite** (прокси для перехвата запросов) + ручной анализ. Автоматические сканеры (ZAP, nikto) дают быстрый обзор, но настоящие находки — в ручной работе.

```bash
# Разведка структуры тестового веб-приложения (своего!)
gobuster dir -u http://192.168.56.101 -w /usr/share/wordlists/dirb/common.txt

# Определить технологии сайта
whatweb http://192.168.56.101
```

**Задание:** поднимите **DVWA** или **OWASP Juice Shop** в лаборатории и пройдите вводные уровни. Для каждой найденной проблемы запишите, какой пункт OWASP Top 10 это закрывает и как разработчик её исправил бы.

## 8. Сети

Глубокое понимание сетей отличает специалиста от «кнопкодава».

- **Wireshark / tcpdump** — анализ трафика. Тренируйтесь читать пакеты своего же трафика.
- **Сегментация, VLAN, файрволы** — как сети защищают изнутри.
- **Wi-Fi:** разберите, как устроена аутентификация WPA2/WPA3. Анализ беспроводных сетей проводите **только на своём оборудовании** — чужой Wi-Fi трогать нельзя.

**Задание:** запустите Wireshark и захватите свой HTTP-трафик к тестовой мишени. Найдите в дампе HTTP-запрос и ответ, разберите заголовки.

## 9. Пароли

Понимание, как хранятся и взламываются пароли, нужно, чтобы строить стойкую аутентификацию.

- **Хеши и соль** — почему пароли нельзя хранить открытым текстом.
- **john**, **hashcat** — восстановление паролей по хешам (на своих тестовых хешах).
- **Словари** — `/usr/share/wordlists/` (например, rockyou). Используются для оценки стойкости.

**Задание:** создайте несколько тестовых хешей разной сложности и попробуйте восстановить их через `john`. Сделайте вывод, какая длина и сложность пароля делает перебор бессмысленным.

## 10. Отчётность и развитие

### Отчёт — это продукт работы

Хороший отчёт понятен и техническому специалисту, и менеджеру: краткое резюме, список находок с уровнем риска, шаги воспроизведения и конкретные рекомендации.

### Сертификации (ориентир для роста)

- **CompTIA Security+** — фундамент по безопасности.
- **CEH** — обзорная по этичному хакингу.
- **OSCP** — практический экзамен, высоко ценится работодателями.

### Что НЕ делает этичный специалист

- Не трогает системы без письменного разрешения — даже «просто посмотреть».
- Не выходит за рамки scope.
- Не разрушает данные и не мешает работе сервисов.
- Не сливает найденные данные — сообщает владельцу (responsible disclosure).

## 13. Повышение привилегий (Privilege Escalation)

Получить доступ к системе — половина дела; дальше идёт повышение прав до root/администратора. Это ключевой навык, и он же показывает администратору, **что нужно закрыть**.

**Linux PrivEsc — что проверяют:**

- Неправильные права на файлы и **SUID/SGID**-бинарники.
- Учётные данные в конфигах, истории команд, переменных окружения.
- Уязвимые задачи **cron**, запущенные от root.
- Возможности sudo (`sudo -l`) и небезопасные `sudoers`.
- Устаревшее ядро и сервисы.

```bash
# Что мне разрешено через sudo
sudo -l

# Поиск SUID-бинарников (потенциальный вектор)
find / -perm -4000 -type f 2>/dev/null

# Автоматизированный аудит (на СВОЕЙ учебной машине)
# linpeas.sh / linenum.sh — собирают подсказки для эскалации
```

**Windows PrivEsc:** неправильные права на службы, незакавыченные пути (unquoted service paths), токены, AlwaysInstallElevated. Инструменты-помощники для аудита: **winPEAS**, **PowerUp**.

> Защита: принцип наименьших привилегий, регулярные обновления, аудит SUID и sudoers, отказ от хранения секретов в конфигах.

**Задание:** возьмите учебную машину с TryHackMe/VulnHub, в которой есть мисконфигурация SUID или sudo, найдите путь к root **вручную** (без автоскриптов), затем опишите в отчёте, как администратор закрыл бы каждый из найденных векторов.

## 14. Active Directory и корпоративные сети

В реальных компаниях почти всё крутится вокруг **Active Directory (AD)**. Понимание AD — то, что отличает джуна от специалиста по корпоративной безопасности.

**Базовые понятия:** домен, контроллер домена, Kerberos, NTLM, group policy, доверительные отношения.

**Типовые техники для изучения (в своей AD-лаборатории):**

- **Enumeration:** **BloodHound** + SharpHound — карта связей и путей к привилегиям.
- **Kerberoasting** и **AS-REP roasting** — извлечение хешей сервисных учёток для оценки стойкости паролей.
- **Pass-the-Hash / Pass-the-Ticket** — почему важно не переиспользовать учётки.
- **Инструменты:** **Impacket** (набор скриптов), **CrackMapExec**, **evil-winrm**.

**Лаборатория:** разверните мини-AD из контроллера домена (Windows Server) и пары клиентов в изолированной сети. Готовые уязвимые наборы: **GOAD** (Game of Active Directory), **DetectionLab**.

> Защита: сложные пароли сервисных учёток, LAPS, мониторинг аномалий Kerberos, тиринг администраторов, отключение устаревших протоколов.

**Задание:** в своей AD-лаборатории постройте граф через BloodHound и найдите кратчайший путь от обычного пользователя до Domain Admin. Опишите, какая мисконфигурация делает этот путь возможным и как её устранить.

## 15. Пост-эксплуатация и pivoting

После получения доступа к одной машине специалист оценивает **масштаб риска**: куда можно пройти дальше.

- **Pivoting** — использование скомпрометированного хоста как точки входа в недоступную напрямую подсеть (туннелирование через **SSH**, **chisel**, **ligolo-ng**).
- **Сбор данных** — где лежат учётки, токены, конфиги (для оценки, что утекло бы).
- **Закрепление (persistence)** — как злоумышленник остался бы в системе (важно понимать, чтобы это обнаруживать и удалять).

```bash
# Пример проброса локального порта через SSH (туннель в недоступную подсеть)
ssh -L 8080:10.10.20.5:80 user@pivot-host

# Динамический SOCKS-прокси через SSH (для сканирования за пивотом)
ssh -D 1080 user@pivot-host
```

> Защита: сегментация сети, мониторинг латерального движения, EDR, ограничение исходящих соединений.

**Задание:** в лаборатории из трёх машин (Kali → пивот → закрытая мишень) пробросьте туннель через пивот и просканируйте закрытую подсеть. Зафиксируйте в отчёте, какая сетевая сегментация предотвратила бы это.

## 16. Продвинутая веб-эксплуатация

Когда базовый OWASP Top 10 освоен, идут более тонкие классы уязвимостей:

- **SSRF** (Server-Side Request Forgery) — заставить сервер обратиться к внутренним ресурсам.
- **Insecure Deserialization** — выполнение кода через десериализацию.
- **JWT-атаки** — слабые секреты, `alg: none`, путаница алгоритмов.
- **Race conditions** — состояние гонки в бизнес-логике.
- **GraphQL / API-безопасность** — introspection, broken object level authorization (BOLA).

**Инструменты:** **Burp Suite** (расширения из BApp Store), **ffuf** для фаззинга, **nuclei** для шаблонного сканирования (по своим целям).

## 21. Реверс-инжиниринг и анализ бинарных файлов

Чтобы понимать, как работают уязвимости и вредоносное ПО, нужно уметь читать программы на уровне машинного кода. Реверс-инжиниринг — это исследование скомпилированных программ без исходного кода: как устроена логика, где хранятся данные, как принимаются решения. Этот навык лежит в основе поиска 0-day уязвимостей, анализа малвари и разработки эксплойтов.

**Что изучать:**

- **Ассемблер (x86/x64, ARM):** регистры, стек, вызовы функций, соглашения о вызовах. Без понимания стека невозможно разобрать переполнение буфера.
- **Статический анализ:** разбор бинарника без запуска — **Ghidra**, **IDA Free**, **radare2/cutter**, **Binary Ninja**.
- **Динамический анализ:** наблюдение за программой в работе — отладчики **gdb (с pwndbg/GEF)**, **x64dbg**, трассировка системных вызовов через `strace`/`ltrace`.
- **Форматы файлов:** структура **ELF** (Linux) и **PE** (Windows), секции, таблицы импорта/экспорта.

> Начинайте с простых crackme-задач: они безопасны, легальны и развивают чутьё без риска.

**Задание:** возьмите простой crackme с crackmes.one или решите reversing-задачу на picoCTF/HackTheBox в своёй лаборатории: найдите проверку пароля в дизассемблере и опишите логику алгоритма.

## 22. Эксплуатация бинарных уязвимостей (основы)

Когда программа неправильно работает с памятью, атакующий может перехватить управление выполнением. Это фундамент low-level безопасности. Изучать это нужно **только на специально подготовленных уязвимых программах** в своёй лаборатории.

**Ключевые классы уязвимостей:**

- **Stack buffer overflow:** переполнение буфера на стеке и перезапись адреса возврата — классика, с которой всё начиналось.
- **Защитные механизмы:** **ASLR**, **DEP/NX**, **stack canaries**, **PIE** — почему они усложняют эксплуатацию и как их проверять (`checksec`).
- **ROP (Return-Oriented Programming):** обход DEP через цепочки гаджетов — идея повторного использования уже имеющегося кода.
- **Format string и heap-уязвимости:** use-after-free, double-free — более сложные темы для следующего этапа.
- **Инструменты:** **pwntools** (Python-фреймворк для эксплойтов), **ROPgadget**, **one_gadget**, **pwndbg**.

> Защита: безопасные функции работы со строками, компиляторные флаги (`-fstack-protector`, `-D_FORTIFY_SOURCE`), языки с безопасной работой с памятью (Rust, Go).

**Задание:** в своёй лаборатории скомпилируйте уязвимую C-программу с отключёнными защитами, вызовите переполнение буфера и перехватите управление. Затем включите защиты поочерёдно и опишите, как каждая из них ломает эксплойт.

## 23. Беспроводные и сетевые атаки

Безопасность сети начинается на канальном и физическом уровнях. Понимание атак на Wi-Fi и локальную сеть помогает грамотно строить защиту. Всё ниже — **только для своей сети или с письменным разрешением владельца**.

**Основные направления:**

- **Wi-Fi (WPA2/WPA3):** перехват handshake и оценка стойкости пароля — **aircrack-ng**, **hcxtools**. Режим мониторинга и совместимые адаптеры.
- **Атаки на канальном уровне:** ARP-spoofing, МИТМ, DHCP-спуфинг, VLAN hopping — почему важна сегментация.
- **Анализ трафика:** **Wireshark** и **tcpdump** — чтение пакетов, поиск утечек и открытых учёток в незашифрованных протоколах.
- **Безопасность протоколов:** почему HTTP/Telnet/FTP опасны и как TLS/SSH решают эти проблемы.

> Защита: WPA3 и сильные пароли, шифрование трафика, сегментация VLAN, защита от ARP-spoofing (DAI), отказ от устаревших протоколов.

**Задание:** перехватите handshake со своей точки доступа и проверьте стойкость своего пароля 
по словарю. Сделайте вывод о том, насколько быстро он подбирается, и усильте его.

## 24. Прикладная криптография

Криптография — это основа всей современной безопасности. Специалисту не нужно изобретать алгоритмы, но критически важно понимать, как их правильно применять и где возникают ошибки.

**Что нужно понимать:**

- **Симметричное и асимметричное шифрование:** AES против RSA/ECC — где что применяется и почему.
- **Хеши и пароли:** почему MD5/SHA1 устарели, зачем нужны **bcrypt/argon2** и соль. Инструменты оценки — **hashcat**, **John the Ripper**.
- **TLS и PKI:** как работают сертификаты, цепочки доверия и чем опасен самоподписанный сертификат в продакшене.
- **Типовые ошибки:** самописная криптография, повторное использование nonce/IV, хранение ключей в коде, слабые источники случайных чисел.

> Золотое правило: никогда не пишите свою криптографию — используйте проверенные библиотеки (libsodium, штатные крипто-модули языка).

**Задание:** возьмите набор хешей паролей, сгенерированных вами самими, и сравните скорость подбора MD5 против bcrypt через hashcat. Сделайте вывод о выборе алгоритма хеширования.

## 25. Безопасность мобильных приложений (Android)

Мобильные приложения — огромная и часто недооценённая поверхность атаки: они хранят токены, ключи и персональные данные прямо на устройстве и активно общаются с бэкендом. Анализ мобильной безопасности — востребованное направление, и изучать его нужно **только на своих или специально учебных приложениях** в эмуляторе.

**Что изучать:**

- **Архитектура APK:** структура пакета (`AndroidManifest.xml`, `classes.dex`, ресурсы), модель разрешений, компоненты (Activity, Service, Broadcast Receiver, Content Provider).
- **Статический анализ:** распаковка и чтение кода — **apktool**, **jadx** (декомпиляция в Java), **MobSF** (автоматизированный аудит APK).
- **Динамический анализ:** перехват трафика приложения через прокси (**Burp Suite** + установка своего CA), инструментирование во время работы через **Frida** и **Objection** (обход root-детекта, SSL-pinning на учебных целях).
- **Типовые проблемы:** хранение секретов и токенов в коде/ресурсах, небезопасное локальное хранилище, экспортированные компоненты, слабая проверка TLS, утечки в логах.
- **Эталон требований:** **OWASP MASVS** и чек-лист **OWASP MSTG** — стандарт того, как проверять мобильное приложение.

> Защита: не храните секреты в приложении, используйте Android Keystore, включайте certificate pinning, не экспортируйте лишние компоненты, чистите чувствительные данные из логов.

**Лаборатория:** учебные уязвимые приложения — **DIVA**, **InsecureBankv2**, **OWASP GoatDroid** — ставятся в эмулятор (Android Studio AVD) или Genymotion.

**Задание:** установите в эмулятор учебное уязвимое приложение, декомпилируйте его в **jadx** и найдите захардкоженный секрет или ключ. Затем перехватите его трафик через Burp и опишите, какие из проблем закрыл бы пункт OWASP MASVS.

## 26. Практикум: как работать с инструментами Kali (мини-курс)

Этот раздел — пошаговый мини-курс по работе с базовыми инструментами Kali на практике. Все команды выполняются **только в своей лаборатории** (раздел 4) против учебной мишени (например, Metasploitable 2 с IP `192.168.56.101`). Идея курса: для каждого инструмента — *зачем он нужен → как запустить → что смотреть в выводе → как защититься*.

### Урок 1. nmap — разведка и сканирование

`nmap` — отправная точка любой работы: он показывает, какие хосты живы и какие сервисы на них открыты.

```bash
# 1. Найти живые хосты в своей подсети (ping-скан, без сканирования портов)
nmap -sn 192.168.56.0/24

# 2. Просканировать порты, версии сервисов и базовые скрипты на мишени
nmap -sV -sC -p- 192.168.56.101

# 3. Сохранить результат во все форматы сразу (пригодится для отчёта)
nmap -sV -oA recon_target 192.168.56.101
```

**Что смотреть:** колонку `STATE` (open/filtered/closed), колонку `SERVICE` и особенно `VERSION` — устаревшие версии и есть главные кандидаты на проверку.

> Защита: закрывайте лишние порты, обновляйте сервисы, прячьте баннеры версий, ограничивайте доступ файрволом.

### Урок 2. Изучаем сервис и ищем известные проблемы

Найденную версию сервиса проверяем по базе эксплойтов **локально** через `searchsploit` (офлайн-копия Exploit-DB).

```bash
# Поиск по названию и версии сервиса из вывода nmap
searchsploit vsftpd 2.3.4

# Посмотреть подробности и путь к описанию
searchsploit -x exploits/unix/remote/17491.rb
```

**Что смотреть:** совпадает ли версия, тип уязвимости (RCE, DoS), есть ли готовый PoC. Это шаг анализа, а не «запустить и сломать».

### Урок 3. Веб-разведка — gobuster и whatweb

Если на мишени есть веб-сервер, изучаем его структуру.

```bash
# Определить, на чём работает сайт
whatweb http://192.168.56.101

# Перебрать скрытые директории и файлы по словарю
gobuster dir -u http://192.168.56.101 -w /usr/share/wordlists/dirb/common.txt
```

**Что смотреть:** коды ответов (`200`, `301`, `403`), найденные админки и резервные файлы. Каждую находку фиксируйте для отчёта.

> Защита: убирайте дефолтные и резервные файлы, настраивайте правильные коды ответов, не раскрывайте стек технологий.

### Урок 4. Анализ трафика в Wireshark

Чтобы понимать, что реально летит по сети, разберите свой же трафик.

1. Запустите Wireshark, выберите интерфейс внутренней сети.
2. Обратитесь к тестовой мишени по HTTP из браузера.
3. Поставьте фильтр `http` и найдите свой запрос и ответ.
4. Разберите заголовки: метод, `Host`, `User-Agent`, `Set-Cookie`.

**Вывод урока:** в открытом HTTP логины и куки видны как есть — отсюда понятно, зачем нужен HTTPS.

### Урок 5. Оценка стойкости паролей — john

Работаем **только со своими тестовыми хешами**, чтобы понять, как защищать аутентификацию.

```bash
# Сгенерировать тестовый хеш (для учёбы)
echo -n 'password123' | md5sum

# Положить хеши в файл hashes.txt и запустить перебор по словарю
john --format=raw-md5 --wordlist=/usr/share/wordlists/rockyou.txt hashes.txt

# Показать восстановленные пароли
john --show --format=raw-md5 hashes.txt
```

**Что смотреть:** насколько быстро подбирается слабый пароль. Вывод — почему нужны длинные пароли, соль и медленные хеши (bcrypt/argon2).

### Как закрепить

Пройдите эти пять уроков по своей мишени подряд и оформите мини-отчёт по шаблону «находка → доказательство → риск → рекомендация» (раздел 6). Когда связка `nmap → анализ → веб → трафик → пароли` станет привычной — переходите к площадкам из раздела 11 (TryHackMe, PortSwigger Academy).

**Задание:** выберите одну открытую службу на своей мишени, пройдите по ней весь цикл (разведка → изучение версии → проверка по searchsploit → вывод о риске) и опишите в конспекте, какие три настройки администратор изменил бы, чтобы её закрыть.
